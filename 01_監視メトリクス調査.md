# 1. EC2 / Spot イベント
EventBridge から取得する  
- EC2 Instance Rebalance Recommendation
- EC2 Spot Instance Interruption Warning
- EC2 instance state change: pending / running / shutting-down / terminated

# 2. Kubernetes イベント
CloudWatch Logs に集約する  
成功イベントだけでなく FailedScheduling 取得し、
「Pod 再配置に失敗したケース」を見ることが重要  
- Node Ready
- Node NotReady
- Pod Killing
👉 正常終了処理中（graceful）
- Pod Scheduled  
👉 このPodをこのNodeで実行することが決定した状態
- Pod FailedScheduling
- Pod Started
- Pod Ready
- Pod Evicted
👉 Pod が Node に強制終了された状態 (Spot終了、OOM、drainなど異常終了寄り)

# Spot 停止時の EKS フローイメージ

`[EC2]`   
Rebalance Recommendationを受信 **（届かない可能性もある）**  
↓  
**（Capacity Rebalancingが有効な場合）**  
新Nodeの起動を試みる **（成功するとは限らない）**  
↓  
Interruption Noticeを受信（Spot停止2分前）  
  
`[K8s Node]`  
↓  
cordon（スケジュール停止、新規Podのデプロイを受け付けない）  
↓  
drain（既存Pod退避）  
  
`[K8s Pod]`   
↓  
Killing（graceful shutdown）  
or  
Evicted（強制 shutdown）  
↓  
（Nodeが空になる）  
  
`[EC2]`   
↓  
Instance shutdown  
  
[K8s Node]  
↓  
NotReady  
  
`[新Node]`   
↓  
instance launch  
**（事前起動できていればこの時点では既に起動済み）**  
↓  
Node Ready  
  
`[K8s Pod]`   
↓  
Scheduled → Started → Ready

# 重要ポイント
## 🟠 Capacity Rebalancing は 「新しいNodeを先に起動する」ための仕組みである
Node だけ事前起動しておき、Pod は Rebalance 時点では動かない  
Rebalance Recommendation を検知して Pod を drain させるには、以下のようなカスタム実装が必要になる
- Node Termination Handler
- Karpenter

## 🟠 Rebalance Recommendation が来ないケースは普通にある
その場合は、いきなり Interruption Notice 受信（Spot停止2分前）となり、新 Node の事前起動はされない  

## 🟠 Rebalance Recommendation が来ても、Interruption Notice の直前なこともある
Rebalance Recommendation 受信  
↓（30秒後）  
Interruption Notice 受信  

この場合も結局 Spot停止（ほぼ）2分前勝負になる
